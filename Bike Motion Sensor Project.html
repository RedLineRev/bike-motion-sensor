// PART 2 · DAILY GIT REFERENCE

The Mental Model

Git tracks history. Three places code lives:

[your folder] → [staging area] → [local repo] → [GitHub] git add git commit git push
  • Your folder is where you edit files.
  • Staging area is where you mark files as ready to save without saving them yet.
  • Local repo is the saved snapshot on your computer.
  • GitHub is the saved snapshot in the cloud.

Daily rhythm: git add → git commit → git push.

The Daily Commands

git status
What changed? Run constantly. Free, prevents mistakes.
git add .
Stage everything. The dot means "every changed file".
git commit -m "..."
Save a snapshot. Write what you did. No "wip" or "stuff".
git push
Upload to GitHub. After your first push, this is all you need.
git log --oneline
What have I done? Recent commits, one per line.
git diff <file>
What changed in this file? When you cannot remember.
git restore <file>
Undo my changes to this file. Throws away changes since the last commit. Be sure.

Good vs Bad Commit Messages

// GOOD
ICM-20948 reads raw accel + gyro on serial monitor Add SD card binary log writer at 200 Hz Mount: bottle cage bracket v2 with EPDM gasket
Future-you knows what changed without reading the code.
// BAD
stuff asdf fixed it wip
When you are hunting for "the version that worked yesterday", you have no idea which commit had what.
// PART 3 · SECRETS

Your repo is public. Anyone can read what you push.

// NEVER COMMIT Wi-Fi passwords, API keys (future phases), personal info like home address or phone numbers, anything that looks like a token.

In Phase 1 you do not have any secrets yet. Just keep the habit: .gitignore blocks .env and config.h files. If you ever add real credentials, they live in those files and never get committed.

The Habit That Saves You

Commit early, commit often. Every time something works that did not work before, commit. Working state to commit. Always.

When something breaks, you want to roll back to "10 minutes ago when it worked", not "this morning before I changed everything".

// 05 · Firmware

What runs on the ESP32.

Small program. Big responsibility. Read the IMU 200 times per second, write each sample to the SD card without dropping anything, and survive a bumpy ride without crashing or running out of memory.

// THE LOOP, IN PLAIN ENGLISH

  • On boot: initialize the IMU over I²C, initialize the SD card over SPI, open a new log file with a name like ride_001.bin.
  • If the user button is pressed and logging is off, start logging. If pressed again, stop logging and close the file.
  • While logging: every 5 milliseconds (200 Hz), read accel x/y/z, gyro x/y/z, magnetometer x/y/z, grab a microsecond timestamp, write a binary record to a small RAM buffer.
  • When the buffer is full enough, flush it to SD. Flushing is slow so it has to happen in a way that does not block sampling.
  • Blink the status LED according to state: solid is powered, slow is logging, fast is SD error or no card.

// THE STACK

  • Framework: Arduino on top of PlatformIO. Most examples and libraries are written for Arduino on ESP32.
  • IMU library: Adafruit ICM20X from the PlatformIO Library Manager. Wraps the ICM-20948 registers in a clean C++ API.
  • SD library: SdFat by Bill Greiman. Faster than the built-in SD.h when you are pushing 200 Hz binary writes.
  • Timing: A hardware timer interrupt at 200 Hz pushes "time to sample" events. A FreeRTOS task on Core 1 does the IMU read. A second task on Core 0 owns the SD flushes. The ESP32-S3 has two cores; we use both.
  • Buffer: A ring buffer in RAM, sized for a few hundred records. If the SD card stalls for a moment, the buffer absorbs it.

// THE BINARY LOG FORMAT

Each record is fixed-size. Easy to parse in Python. Easy to count.

struct Sample { uint32_t t_us; // microseconds since boot int16_t ax, ay, az; // raw accelerometer counts int16_t gx, gy, gz; // raw gyroscope counts int16_t mx, my, mz; // raw magnetometer counts uint8_t flags; // bit 0: button pressed, bit 1: SD error uint8_t reserved; // padding to 24 bytes (aligned) };

24 bytes per sample × 200 Hz = 4.8 KB/sec = about 17 MB per hour of ride. A 16 GB card holds nearly 1000 hours. Storage is not the bottleneck.

// WHY RAW COUNTS, NOT FLOATS

  • Half the bytes. Twice as fast to write.
  • No precision loss. You convert to g and degrees-per-second in Python, where math is free.
  • If you ever change the IMU's range setting, the raw counts still mean the same thing within their range. The scale factor moves to a header field.
Q · Where Does The IMU Range Setting Live?

Accel can be ±2g, ±4g, ±8g, ±16g. Gyro can be ±250, ±500, ±1000, ±2000 dps. For MTB jumps and drops, accel impacts can hit 8g or more. Gyro spin during whips or hard berms can hit 500 dps. Start at ±8g and ±1000 dps. Document this in your firmware in a comment block at the top.

Q · How Do You Know Samples Are Not Being Dropped?

Every sample carries a microsecond timestamp. In Python, after loading, compute the time delta between consecutive samples. If a delta is much greater than 5000 microseconds, that gap is a dropped sample. A healthy log has greater than 99% of deltas in a tight band around 5000 µs.

Q · What Happens If The Card Pulls Out Mid-Ride?

Bad. The file is unclosed and the last few KB of data are gone. Mitigation: lock the card mechanically into the slot with the enclosure design. Belt-and-suspenders: flush the file every 1 second so an unclean shutdown loses at most 1 second of data.

Build it in five chunks: (1) blink the LED, (2) read the IMU and print to serial, (3) write a hello string to the SD card, (4) combine: read IMU and write CSV to SD, (5) upgrade to binary + timer + dual-core. Each is a commit. Each is a session.

// 06 · Data Capture & Sync

Getting clean labeled data is most of the project.

Phase 1 lives or dies on data quality. If the data is clean, labeled, and synced to video, Phase 2 is easy. If it is not, Phase 2 is impossible.

The Sync Protocol

Every ride starts and ends with the same ritual. Burn it into muscle memory.

// THE RIDE START / END RITUAL

Same five steps. Every ride.

  1. Power on the device. Confirm LED is solid. SD card is inserted.
  2. Mount the device. Bottle cage bolts. EPDM gasket between case and frame. Tighten by hand, do not crank.
  3. Start the GoPro recording. Confirm the red dot.
  4. Three sharp taps on top of the device while the GoPro is filming the bike. Spaced about half a second apart. Hard enough to spike the accelerometer.
  5. Push the log button on the device. LED shifts to slow blink.

At the end of the ride: stop the log button, three more taps for the end sync, stop the GoPro. The taps at the end are belt-and-suspenders. If the IMU and GoPro clocks drift, you can correct from either end.

Why The Sync Taps Work

  • Each tap shows up as a clean accel spike of several g, visible in the data as a tall narrow pulse.
  • The same tap is audible in the GoPro track and visible if you can see the bike.
  • You convert the tap event to a frame number in the GoPro and a sample number in the IMU log. Subtract: you have the time offset.
  • Once the offset is known, you can label tricks in the video and translate the frame number to an IMU sample range.

The Python Workflow

// LOADER

A function load_ride(path) that reads the binary file and returns a pandas DataFrame with columns: t_us, ax, ay, az, gx, gy, gz, mx, my, mz, flags. Add a derived column t_s in seconds since boot.

Bonus: a second function that converts raw counts to physical units. ±8g range means each count is 8/32768 ≈ 0.000244 g. ±1000 dps range means each gyro count is 1000/32768 ≈ 0.0305 degrees/sec. Magnetometer scale is in the ICM-20948 datasheet.

// PLOTTING NOTEBOOK

A Jupyter notebook that:

  • Loads a ride file with the loader function above.
  • Plots accelerometer (3 axes) as a stacked time series across the whole ride.
  • Plots gyroscope (3 axes) below it.
  • Plots magnetometer (3 axes) below that.
  • Has zoom controls so you can isolate a 5-second window and see one trick clearly.
  • Computes summary stats: ride duration, sample count, dropped-sample count, max accel magnitude, max gyro magnitude.

// LABELING HELPER

Open the GoPro file alongside the IMU plot. Watch the run. When you see a feature happen, note the GoPro timestamp and a short label like drop_1m, jump_small, berm_left, bunny_hop. The notebook writes these into a labels.csv next to the ride file. The CSV columns: ride_id, t_start_s, t_end_s, label, notes.

Phase 2 will train on this CSV. Phase 1 just makes sure we can produce it cleanly.

What Each Feature Looks Like In The Data

Before you ever ride, build a mental model of what each event should look like. After your first ride, check whether reality matches.

SIGNATURE · DROP

A short freefall where accel z drops near zero, followed by a sharp positive accel spike on landing (could be 4 to 10 g depending on drop height and suspension). Gyro stays roughly quiet through the air, may pick up on landing.

SIGNATURE · JUMP

Distinct ramp-up of pitch from compressing into the lip, freefall while airborne (accel z near zero again), then a softer landing spike than a drop because the rider rotates the bike to match the landing slope.

SIGNATURE · BERM

Sustained accel in the lateral axis (y in our frame: forward is x, left is y, up is z) for one to several seconds while the bike rounds the berm. Roll angle from the gyro should integrate to a banked attitude that holds steady through the turn.

SIGNATURE · BUNNY HOP

Quick pitch up from rear wheel lift, brief airtime (much shorter than a real jump), then pitch back down. Smaller accel signatures than a drop because there is no ramp and no big landing impact.

SIGNATURE · ROCK GARDEN

High-frequency noise across all three accel axes. Magnitude can be high but the spikes are short. Distinctive in the time-domain plot, and very distinctive in a spectrogram.

  • Do not keep going when you are frustrated. Walk away for 15 minutes. Half the time the answer is obvious after a break.

// 09 · When Stuck

You will get stuck. Here is the playbook.

Every real project has moments where nothing works and it feels like you will never finish. That is normal. Engineers spend more time stuck than building. The difference is they have a routine. Here is yours.

// THE 30-MINUTE PROTOCOL

Stuck for 30 minutes? Run this.

  1. Write down what you are seeing. Not in your head. On paper or in a doc. Symptoms, error messages, what changed since it last worked.
  2. Write down what you tried. Including things that did not help. Stops you from going in circles.
  3. Write down what you expect to happen, and why. Often the bug is in your assumptions, not your code.
  4. Now talk to Claude. Use the bug report template. Half the time you figure it out while writing the prompt.
  5. If Claude's first answer does not fix it, tell it what you tried and what happened. Do not just paste the same prompt with a new error.
  6. Still stuck after 60 total minutes? Stand up. Drink water. Walk around. Come back fresh. If still stuck after that, see the escalation ladder below.

Common Bugs You Will Probably Hit

Click each one to see typical causes and what to check first.

IMU ICM-20948 not detected on I²C (i2cdetect shows nothing) ▸
Likely Causes
  • Stemma QT cable is unseated. Click it in until it snaps.
  • Wrong I²C bus selected in code (ESP32-S3 has multiple).
  • Power not reaching the board. The IMU board needs 3.3V on Vin and GND.
  • Address mismatch. ICM-20948 default address is 0x69. Many examples assume 0x68.
What to Check First
  1. Run an I²C scanner sketch. Watch the serial monitor for the detected address.
  2. Reseat both ends of the Stemma QT cable.
  3. Confirm the Feather's Stemma QT port is powered (LED on the Feather is on).
  4. Update the library example to use address 0x69 explicitly.
SD SD card init fails or writes return 0 bytes ▸
Likely Causes
  • Wrong CS (chip select) pin. SdFat default may not match how you wired it.
  • Card not formatted as FAT32. Most are out of the box, some are not.
  • SPI clock too high for the breakout's wiring. Try lowering to 4 MHz.
  • Card is fake or counterfeit. Cheap Amazon cards lie about capacity and corrupt under load.
What to Check First
  1. Reformat the card as FAT32 on your PC.
  2. Make sure the CS pin in code matches the wire. Print the value in setup() to be sure.
  3. Try SdFat at SPI_HALF_SPEED first. Increase later.
  4. Test with a different card. Brand-name SanDisk or Samsung is worth the few extra dollars.
IMU Samples are at the wrong rate or non-uniform ▸
Likely Causes
  • You are using delay(5) instead of a hardware timer. Delays are not precise.
  • SD writes are blocking the sample loop. Buffer is too small or flush is on the wrong core.
  • I²C clock is the default 100 kHz. Read at 200 Hz with 9 axes takes longer than 5 ms at that speed. Bump to 400 kHz.
What to Check First
  1. Plot the time deltas in Python. Look for a tight band at 5000 µs or a messy spread.
  2. Use a hardware timer interrupt for sampling, not delay().
  3. Set Wire.setClock(400000) after Wire.begin().
  4. Move SD flushes to Core 0 and IMU reads to Core 1.
FIRMWARE ESP32-S3 reboots randomly or hangs after a few seconds ▸
Likely Causes
  • Stack overflow in a FreeRTOS task. Default stack is small. Sampling and SD tasks need a few KB each.
  • Watchdog timer firing because a task is starving. Add vTaskDelay(1) at end of loop bodies.
  • Brownout from a current spike when the SD card writes. Battery sagging or USB cable too thin.
What to Check First
  1. Enable core dump output and look at the panic reason in the serial monitor right before reboot.
  2. Increase task stack size to 8 KB and see if it changes.
  3. Try a different USB cable. Try a different USB port or a wall adapter.
  4. Add a 470 µF capacitor across the SD breakout's Vin and GND.
POWER Battery dies way faster than expected ▸
Likely Causes
  • Wi-Fi or BLE radios are powering up even though you do not use them. Disable both explicitly in setup().
  • Status LED is full-brightness instead of dim. Move from 330 Ω to 1 kΩ resistor or use PWM.
  • SD writes are spiking current. Larger buffer = fewer flushes.
  • Battery is old or low capacity. Genuine Adafruit 1200 mAh should give you 5+ hours of logging.
What to Check First
  1. Verify WiFi.mode(WIFI_OFF); btStop(); in setup().
  2. Measure current draw with a multimeter in series on the battery lead. Should be under 100 mA logging.
  3. Charge the battery fully before each session and note the voltage.
PYTHON Binary log loads but plots look like noise or are mis-scaled ▸
Likely Causes
  • You forgot to convert from raw counts to physical units. ±8g range means each count is about 0.000244 g.
  • Endianness mismatch. ESP32 is little-endian; numpy reads need to use '<' in the dtype.
  • Struct alignment in C++ added invisible padding bytes. Use __attribute__((packed)) on the Sample struct.
  • You labeled the axes wrong relative to the mount. X may be left-right instead of forward-back.
What to Check First
  1. Print one record in hex from the firmware and compare to what Python reads. They must match byte-for-byte.
  2. Apply scale factor and re-plot.
  3. While bench-testing, push the device along each axis on purpose. Confirm the right axis sees the motion.
MOUNT Enclosure rattles loose or rotates mid-ride ▸
Likely Causes
  • EPDM gasket too thick. Bolts compress unevenly and let it walk.
  • Bolt threads bottomed out before clamping the bracket.
  • 3D print warped at the bolt holes. Layer adhesion failed under load.
  • Bottle cage bolts are only meant for a few pounds of pull. Trail vibration adds shock load.
What to Check First
  1. Reduce gasket to 1 mm. Even compression matters.
  2. Add a backup zip-tie around the enclosure and the down tube as a safety leash.
  3. Reprint the bracket in PETG with thicker walls (3 mm minimum near bolt holes).
  4. Check bolt torque after every ride for the first three rides.
